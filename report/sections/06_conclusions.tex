\chapter{Conclus\~oes}

\section{Resumo do trabalho realizado}

O projeto BaseBallX teve como objetivo demonstrar a aplica\c{c}\~ao pr\'atica de t\'ecnicas de Web Sem\^antica sobre um conjunto de dados hist\'oricos de baseball. Partindo de vinte ficheiros CSV provenientes do Lahman Baseball Database, foi constru\'ido um pipeline completo que transforma os dados em RDF, os armazena num triple store GraphDB e os disponibiliza numa aplica\c{c}\~ao Web Django para consulta, navega\c{c}\~ao e análise.

O resultado final \'{e} uma plataforma que permite explorar jogadores, equipas, franquias, sal\'arios, pr\'emios, Hall of Fame e resultados de p\'os-temporada atrav\'es de consultas SPARQL. A interface cobre desde listas de cat\'alogo at\'e p\'aginas de detalhe, compara\c{c}\~ao de jogadores, grafos de rela\c{c}\~oes, um painel de analytics avan\c{c}ado e funcionalidades de administra\c{c}\~ao com autentica\c{c}\~ao.

\section{Objetivos alcan\c{c}ados}

Os principais objetivos propostos foram cumpridos:

\begin{itemize}[leftmargin=*]
  \item \textbf{Convers\~ao RDF}: os vinte ficheiros CSV foram totalmente transformados em triplos RDF com recurso \`a biblioteca \texttt{rdflib}, produzindo um grafo coerente com identifica\c{c}\~ao uniforme de entidades.
  \item \textbf{Armazenamento sem\^antico}: o grafo resultante foi carregado no GraphDB e fica persistentemente dispon\'ivel para interroga\c{c}\~ao via SPARQL.
  \item \textbf{Interface de navega\c{c}\~ao}: foram implementadas vistas de cat\'alogo, detalhe e compara\c{c}\~ao para jogadores e equipas, com dados obtidos diretamente do triple store em tempo de pedido.
  \item \textbf{Analytics avan\c{c}ados}: o painel de analytics agrega, numa \'unica p\'agina com sete separadores, l\'ideres globais de batting, franchise history, os sal\'arios mais altos da hist\'oria, timeline de pr\'emios, Hall of Fame, ranking de managers e uma vista combinada.
  \item \textbf{Visualiza\c{c}\~ao de grafos}: a integra\c{c}\~ao do Cytoscape.js permite explorar visualmente as rela\c{c}\~oes entre jogadores e entidades relacionadas.
  \item \textbf{\'Area de administra\c{c}\~ao}: utilizadores autenticados podem submeter sugest\~oes de corre\c{c}\~ao de dados, com fluxo de aprova\c{c}\~ao pelo administrador.
  \item \textbf{Quiz}: foi inclu\'ido um m\'odulo de quiz sobre dados hist\'oricos, com registo de tentativas por utilizador.
\end{itemize}

\section{Desafios encontrados}

Ao longo do desenvolvimento foram encontradas algumas dificuldades:

\begin{itemize}[leftmargin=*]
  \item \textbf{Qualidade dos dados de origem}: os ficheiros CSV cont\^em valores em falta, inconsist\^encias de encoding e identificadores sem correspondência cruzada, o que exigiu normaliza\c{c}\~ao cuidadosa durante a convers\~ao para RDF.
  \item \textbf{Performance das consultas SPARQL}: algumas consultas que agregam dados de m\'ultiplos grafos (por exemplo, estat\'isticas combinadas de batting e pitching) revelaram lat\^encias not\'aveis. Foi necess\'ario simplificar certas queries e limitar os resultados retornados para manter a interface responsiva.
  \item \textbf{Modelação do grafo RDF}: decidir o n\'ivel de granularidade das entidades RDF -- nomeadamente se as estat\'isticas por época deviam ser recursos pr\'oprios ou literais associados ao jogador -- implicou v\'arias itera\c{c}\~oes sobre o script de convers\~ao.
  \item \textbf{Integra\c{c}\~ao Django--GraphDB}: a aus\^encia de uma camada ORM para triple stores obrigou a escrever e manter manualmente todas as consultas SPARQL, o que aumentou o esfor\c{c}o de desenvolvimento e de teste.
  \item \textbf{Arranque do ambiente}: a depend\^encia do GraphDB Desktop como processo externo tornou o arranque do projeto mais fr\'agil do que numa arquitetura puramente baseada em contentores ou servi\c{c}os geridos.
\end{itemize}

\section{Melhorias futuras}

Com mais tempo de desenvolvimento, as melhorias mais relevantes seriam:

\begin{itemize}[leftmargin=*]
  \item \textbf{Conteineriza\c{c}\~ao}: empacotar a aplica\c{c}\~ao Django e o GraphDB em contentores Docker com um ficheiro \texttt{docker-compose.yml}, eliminando a configura\c{c}\~ao manual do ambiente.
  \item \textbf{Cache de consultas}: introduzir uma camada de cache (por exemplo, Redis) para as queries SPARQL mais frequentes, reduzindo a lat\^encia percebida pelo utilizador.
  \item \textbf{P\'aginas de detalhe adicionais}: estender a navega\c{c}\~ao para cobrir entidades ainda sem p\'agina pr\'opria, como s\'eries de p\'os-temporada, All-Star Games individuais e pr\'emios de managers.
  \item \textbf{Pesquisa avan\c{c}ada}: implementar um formulário de pesquisa com filtros combinados (época, liga, posi\c{c}\~ao, faixa de sal\'ario) totalmente resolvido via SPARQL.
  \item \textbf{Exporta\c{c}\~ao de dados}: permitir ao utilizador exportar os resultados de uma vista ou consulta nos formatos CSV, JSON ou Turtle.
  \item \textbf{Ontologia pr\'opria}: substituir o vocabulário informal atual por uma ontologia formal de dom\'inio, melhorando a interoperabilidade e permitindo racioc\'inio sobre o grafo.
  \item \textbf{Testes autom\'aticos}: alargar a cobertura de testes de integra\c{c}\~ao para incluir as views e queries SPARQL mais cr\'iticas, garantindo maior robustez face a altera\c{c}\~oes no grafo.
\end{itemize}

\section{Considera\c{c}\~oes finais}

O projeto demonstrou que a combina\c{c}\~ao de RDF, SPARQL e um triple store \'{e} uma abordagem vi\'avel e expressiva para explorar conjuntos de dados relacionais complexos. A representa\c{c}\~ao sem\^antica dos dados facilita a navega\c{c}\~ao por rela\c{c}\~oes que numa base de dados relacional exigiriam m\'ultiplas jun\c{c}\~oes, e abre caminho para funcionalidades como infer\^encia e liga\c{c}\~ao a dados externos da Web aberta.

Do ponto de vista acad\'emico, o trabalho permitiu aplicar os conceitos fundamentais de Web Sem\^antica -- modelação RDF, consulta SPARQL e integra\c{c}\~ao num sistema Web -- num contexto pr\'atico e com dados reais, consolidando a compreens\~ao das tecnologias estudadas ao longo da unidade curricular.
